# Claude Desktop-style 竞赛智能工作台定位

## 1. 一句话定位

本项目要做的是面向大学生竞赛项目的 Claude Desktop-style 智能工作台，而不是通用 Claude 克隆。

```text
竞赛是产品核心，Claude Desktop-style 是界面流程和交互范式。
```

系统应该让学生、指导老师和竞赛管理者通过类似 Claude 桌面端的会话流完成导入、审计、问诊、补证、优化计划、二次审计和导出。会话可以处理普通问答，但凡是项目判断、评分和优化，都必须绑定真实竞赛项目材料包、规则结果和证据链。

## 2. 桌面端应该像什么

桌面端界面和流程应效仿 Claude 桌面端的会话工作区，但使用本项目自己的名称、视觉语言、图标、配色、文案和竞赛业务组件。

推荐界面骨架：

- 左侧 Sidebar：会话历史、最近项目、项目空间、规则包、导出记录；
- 中央 Conversation：用户消息、系统计划、工具调用卡片、审计进度、风险解释、补证建议；
- Composer：项目拖拽/附件、快捷命令、发送按钮和常规问答输入；
- Artifact / Project Panel：资产清单、CPIR、声明—证据、规则 finding、修复计划、报告预览和 diff；
- Settings Permission：联网、LLM、导出、读取项目外文件、写入工作区等高风险边界集中设置；
- Status Strip：当前项目、赛道、规则版本、LLM 是否启用、工作区路径和审计状态。

推荐交互流程：

```text
新建会话
→ 拖入或选择竞赛项目材料包
→ 智能体建立安全工作副本并生成审计计划
→ 会话中展示工具调用卡片和审计进度
→ 规则包、证据匹配和一致性检查全方位匹配
→ 只输出 blocker/warning、可信债务、补证任务等缺点
→ 询问用户是否优化项目
→ 用户继续追问某个风险项
→ 系统基于证据链解释原因并生成补证优先级
→ 用户确认后生成安全工作区优化方案或报告
→ 二次导入修复后项目并比较差分
```

这意味着会话是主交互形态，但会话内容必须围绕竞赛项目、审计证据和补证任务展开。

## 3. Claude Desktop-style 能参考什么

可以参考以下模式：

| 模式 | 本项目落点 |
|---|---|
| Agent Loop | 受控 StagedAuditPipeline / AgentRuntime |
| Tool Registry | 注册 inventory、audit、report、repair、diff 等竞赛工具 |
| Permissions | 控制文件写入、联网、LLM、命令执行和项目外读取 |
| Hooks | 审计前后、修复前后、导出前后做安全和完整性检查 |
| Session / Memory | 保存赛道、规则版本、审计输入、风险项和用户确认 |
| Implicit Rule Matching | 按赛道规则包、证据状态和一致性结果隐式匹配，不在界面暴露技能模块 |
| Context Compaction | 将长材料压缩为资产摘要、证据摘要和风险摘要 |
| Diff-first Workflow | 先生成修复计划和差分，不直接覆盖原项目 |
| Permission Modes | 高风险动作必须经过权限模式边界 |

这些模式必须服从竞赛审计流程。界面流程可以高度接近 Claude 桌面端，但业务对象、工具、权限、报告和视觉资产必须属于本项目。

## 4. 不能写成什么

禁止把 MD 写成下面这些方向：

```text
我要做一个类 Claude 的通用 AI 助手。
我要做一个能随便读写项目、执行命令、自动改代码的智能体。
我要做一个无项目上下文的空白聊天应用。
我要复制 Claude 的品牌、图标、配色、文案或专有视觉资产。
我要让 LLM 根据感觉给竞赛项目打分。
我要让 AI 自动补齐商业数据、实验结果、合作证明或专利证明。
我要把所有功能写进 main.cpp、Controller、Manager 或单个大文件。
```

这些写法会让项目脱离竞赛场景，也会破坏可信审计的可解释性。

## 5. 正确的 MD 写法

正确表述应该是：

```text
本项目是面向大学生竞赛项目材料包的智能审计与补证工作台。
它在界面和流程上采用 Claude Desktop-style 的会话工作区、侧边栏、工具调用可视化、设置式权限边界、项目记忆、上下文压缩和 diff-first 修改体验，
但所有能力都围绕竞赛项目材料包展开。

系统核心目标不是陪聊、不是通用代码生成、不是代写项目，而是帮助用户将项目材料转化为可检查、可解释、可复核的证据对象，
发现材料缺口、证据缺口、逻辑矛盾、赛道规则风险和答辩风险，
并输出补证任务、修复计划、二次审计差分和 Markdown/JSON 报告。
```

## 6. 核心用户工作流

### 6.1 学生赛前自查

```text
导入项目材料包
→ 创建或进入项目会话
→ 选择竞赛赛道
→ 运行可信审计
→ 在会话中查看工具卡片、blocker/warning 和报告 artifact
→ 追问风险原因和缺失材料
→ 生成补证任务和修复计划
→ 补充材料后再次审计
→ 导出提交前报告
```

### 6.2 指导老师批量诊断

```text
导入多个项目
→ 统一规则包审计
→ 按 blocker、可信债务和证据覆盖率排序
→ 查看每个项目的高风险声明
→ 输出整改清单
→ 跟踪多轮修复差分
```

### 6.3 竞赛训练营项目打磨

```text
项目初诊
→ 智能体按赛道规则包、证据状态和一致性问题隐式匹配
→ 补证任务拆解
→ 路演和答辩风险问答
→ 最终材料一致性检查
→ 导出可信报告
```

## 7. 模块边界

核心业务必须在 C++ Core 中：

- `loader` 负责导入和解包；
- `inventory` 负责资产识别；
- `text` 负责文本抽取；
- `cpir` 负责项目中间表示；
- `claim` 负责声明抽取；
- `evidence` 负责证据匹配；
- `consistency` 负责一致性审计；
- `rules` 负责规则包加载、校验和执行；
- `audit` 负责可信评分和审计汇总；
- `repair` 负责补证任务、修复计划和 diff；
- `report` 负责 Markdown/JSON 报告。

Agentic Runtime 只负责竞赛审计流程编排和安全交互：

- `ToolRegistry` 注册竞赛工具；
- `PermissionGate` 控制风险动作；
- `LifecycleHookManager` 阻断不安全流程；
- `AuditSessionStore` 保存审计会话；
- `ProjectMemory` 保存项目规则和用户确认；
- JSON 规则包承载赛道专用分析，不在界面暴露技能模块；
- `PermissionGate` 和 Workbench 设置中的权限模式管理高风险能力边界。

QML Controller 只能桥接 UI 和 C++ 服务，不允许实现扫描、规则、评分、证据匹配或报告拼接。

## 8. 会话能力边界

会话工作区允许：

- 解释某个 blocker 为什么触发；
- 追问缺失材料；
- 按赛道给出补证优先级；
- 根据已有审计结果生成答辩风险问题；
- 帮用户理解报告；
- 在用户确认后生成模板或修复计划。

会话工作区禁止：

- 直接决定最终评分；
- 修改原始项目；
- 伪造用户、营收、合作、专利、实验或市场数据；
- 跳过 RuleEngine、EvidenceMatcher 或 TrustScoreCalculator；
- 自由执行 shell 命令；
- 在没有有效 LLM 配置时联网，或上传被敏感文件策略阻止的材料；
- 脱离当前项目、规则和证据链给出项目判断或评分。

## 8.1 混合研判（LLM 先判断、规则校验）

在 endpoint、模型和 API key 组成的 LLM 配置通过校验后，会话工作区自动启用混合研判：

```text
确定性审计完成
→ LLM 基于审计上下文（资产、画像、规则风险、证据状态、补证任务）给出风险研判和评分建议
→ AdvisoryReconciler 用确定性结果逐条校验研判
    · 命中规则风险或证据缺口 → 已印证
    · 找不到确定性依据 → 待核实（仅参考，不影响评分）
    · 与确定性结论矛盾（如判为通过但存在 Blocker）→ 与规则冲突，降级并标注
→ 最终可信评分仍取 TrustScoreCalculator 的确定性结果
```

关键约束：LLM 的评分建议只用于和确定性评分对比解释差异，不作为最终评分；混合研判需要联网和有效 API key；配置缺失、无效或网络不可用时，会话仍以确定性审计为准。

## 8.2 项目访问模式

导入项目时提供四种访问模式，对应 tokenicode/Claude Code-style 的会话模式；这些模式集中放在设置中，不作为主输入栏的默认操作：

- ask（默认）：在隔离项目副本内问答和读取材料，不改原项目；
- plan：只生成执行计划，不执行工具写入或高风险动作；
- code：在受控工作区执行可审计工具，并将产物写入工作区；
- bypass：经用户显式切换后允许读取用户选择的原始项目路径，并可写入隔离工作区；仍不允许修改原项目或执行命令。

所有模式都通过 PermissionGate 校验；bypass 属于高风险动作，不能由模型自行开启。

## 9. 验收标准

如果某个功能声称是 Claude Desktop-style，必须同时证明：

- 它绑定竞赛项目材料包和 AuditSession；
- 它调用的是受控竞赛工具；
- 它经过权限模型和 hooks；
- 它不覆盖原项目；
- 它不让 LLM 决定最终评分；
- 它的输出能追溯到规则 ID、证据来源或明确的缺失材料；
- 它遵守 `doc/12.md` 的模块拆分规则。

只有满足以上条件，Claude Desktop-style 能力才算服务了竞赛主业务。
