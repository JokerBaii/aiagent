# 调研与需求分析文档

## 1. 项目定位

本项目面向大学生创新创业竞赛、商业创新竞赛、科研实践竞赛、工程产品竞赛、软件开发竞赛、社会实践竞赛、电商三创赛、大创项目、课程设计与毕业设计等场景，构建一个面向完整项目材料包的可信编译、证据审计与自动补证系统。

项目不是普通 AI 助手，也不是代码生成器或商业计划书生成器，而是面向完整竞赛项目材料包，提供项目资产识别、材料完整性审计、项目声明—证据匹配、材料一致性检查、赛道规则审计、可信评分、补证任务生成、修复建议和二次审计能力。

项目可以具备 Claude Desktop-style 的会话协作体验，并将会话作为桌面端主操作入口；但这种体验必须服务于竞赛场景：导入项目、解释审计风险、追问缺失材料、调用受控审计工具、生成补证任务、组织修复计划和导出报告。系统不以无项目上下文的通用聊天、通用代码生成或自由执行任务为目标。

系统输入包括但不限于：

- 项目申报书；
- 商业计划书；
- 路演 PPT；
- 市场调研报告；
- 竞品分析表；
- 财务预测表；
- 用户访谈记录；
- 问卷数据；
- 源码仓库；
- README；
- 部署文档；
- 实验数据；
- 论文、专利、软著；
- 合作协议；
- 社会实践证明；
- 演示视频说明和样机照片。

系统输出包括：

- 项目资产清单；
- 项目类型识别结果；
- CPIR 项目中间表示；
- 项目声明列表；
- 声明—证据匹配结果；
- 材料一致性审计结果；
- 赛道规则审计结果；
- 可信评分与可信债务；
- blocker / warning 风险项；
- 补证任务；
- 修复计划；
- 二次审计差分；
- Markdown / JSON 可信报告。

## 2. 背景调研

大学生竞赛项目规模巨大，且项目形态并不局限于代码仓库。教育部公布的中国国际大学生创新大赛（2025）信息显示，该赛事共有 5673 所学校、619 万个项目、2443 万人次报名参赛，并最终产生获奖项目 4729 个。来源：<https://news.eol.cn/yaowen/202602/t20260228_2719981.shtml>

“挑战杯”全国大学生课外学术科技作品竞赛覆盖科研、工程、社会实践和创新应用等多类项目；第十九届“挑战杯”吸引全国 2700 多所高校、40 余万件作品参赛，并设置主体赛、“揭榜挂帅”、“人工智能+”专项赛和项目对接转化会等内容。来源：<https://www.cae.cn/cae/html/main/col125/2025-11/14/20251114102514369261698_1.html>

全国大学生电子商务“创新、创意及创业”挑战赛强调创新、创意和创业实践，面向电商、商业创新、创业实践等方向。来源：<https://www.3chuang.net/>

现有通用 AI 编程工具已经具备较强代码开发能力。例如 GitHub Copilot cloud agent 可以研究仓库、创建实现计划并在分支上修改代码，用户可审查 diff 并创建 pull request。来源：<https://docs.github.com/copilot/concepts/agents/coding-agent/about-coding-agent>

Repomix 可以将代码仓库打包成 AI-friendly 的 XML、Markdown、JSON 或纯文本格式，用于输入大模型。来源：<https://repomix.com/>

Claude Code 相关文档显示，其 Agent SDK 包含 built-in tools、hooks、subagents、MCP、permissions、sessions 等能力。来源：<https://code.claude.com/docs/en/agent-sdk/overview>

这些工具对代码开发或仓库上下文整理有价值，但并不专门处理大学生竞赛中的申报书、商业计划书、路演 PPT、市场调研、竞品分析、财务预测、成果证明、社会实践证明等材料，也不提供面向竞赛规则的可信评分、补证任务和二次审计闭环。

因此，本项目吸收的是通用 agent 产品中的工作流能力，而不是它们的产品定位。竞赛规则、证据链、材料一致性和补证闭环必须排在对话体验之前。

## 3. 目标用户

### 3.1 大学生竞赛团队

他们需要知道项目是否能提交、材料缺什么、商业逻辑是否成立、技术实现是否支撑创新点、答辩时哪里最容易被质疑，以及临近提交前应优先补什么。

### 3.2 指导老师

指导老师面对多个团队项目时，需要快速定位项目风险、判断是否只是概念包装、生成整改建议，并跟踪多轮修改效果。

### 3.3 高校创新创业学院与项目孵化平台

高校创新创业学院、竞赛训练营和项目孵化平台需要批量项目诊断、多赛道规则审计、项目风险分级、补证任务跟踪和版本对比。

### 3.4 课程设计、毕设和大创项目学生

这类用户需要检查报告、代码、README、部署说明、实验数据与成果材料是否一致，辅助课程设计、结题和答辩。

## 4. 核心痛点

### 4.1 项目材料分散，缺少统一资产清点

竞赛项目通常包含 Word、PPT、PDF、Excel、源码、图片、视频、数据、模型、证明材料等多种文件。学生很难判断哪些材料是核心证据，哪些材料只是辅助材料，哪些材料存在风险。

常见问题包括：缺少申报书、缺少商业计划书、缺少市场调研、缺少竞品分析、缺少财务预测、缺少成果证明、缺少 README、缺少部署说明、缺少实验数据、上传无关文件、上传 `.env` 或密钥、项目包包含大量 build/dist/node_modules 等生成物。

### 4.2 项目声明缺少证据支撑

许多项目材料存在“表达强、证据弱”的问题，例如“市场前景广阔”“已有大量用户”“已完成市场验证”“技术领先”“已具备落地能力”，但缺少调研数据、用户访谈、后台截图、订单流水、合作协议、专利受理通知、实验结果或竞品来源。

### 4.3 材料之间存在矛盾

常见矛盾包括：PPT 与申报书目标用户不同，商业模式与财务预测不一致，市场调研对象与用户画像不一致，README 写了功能但代码未实现，申报书写了技术创新但没有实验或代码证据。

### 4.4 商业创新项目缺少商业闭环

商业创新项目经常目标用户模糊、市场规模无来源、竞品分析过浅、收入模式不清晰、成本结构缺失、财务预测无假设、增长路径不可信、风险分析空泛。

### 4.5 技术和科研项目缺少可复核证据

技术工程和科研项目经常缺少系统架构图、核心模块说明、部署说明、实验数据、baseline、评价指标、复现实验脚本、论文/专利/软著证明，导致创新点无法被复核。

### 4.6 临近提交前缺少修复优先级

学生知道项目有问题，但不知道哪些必须先修。系统需要区分 blocker、warning、info、evidence missing、consistency risk、overclaim risk，并给出可执行补证任务。

## 5. 需求目标

总体目标：构建一个面向大学生竞赛项目材料包的可信审计与自动补证系统，实现资产识别、项目类型识别、CPIR 构建、声明抽取、证据匹配、材料一致性审计、赛道规则审计、可信评分、补证任务、修复计划、二次审计和报告导出。

一句话目标：帮助学生把“看起来完整的项目”打磨成“材料完整、逻辑自洽、证据充分、规则匹配、可提交、可答辩、可复核”的可信竞赛项目。

## 6. 系统边界

系统应该做：项目材料包识别、文件格式识别、资产语义分类、竞赛类型识别、规则包匹配、材料完整性审计、声明抽取、证据匹配、材料一致性审计、商业闭环审计、技术支撑审计、科研可复核性审计、社会实践影响力证据审计、可信评分、补证任务、报告导出、修复建议、二次审计，以及围绕这些结果展开的竞赛顾问式对话。

系统不应该做：虚构市场数据、虚构用户数量、虚构营收、虚构合作单位、虚构专利/软著/论文、虚构实验结果、凭空生成不可验证结论、未经用户确认覆盖原项目、把 LLM 输出作为最终裁决依据、默认上传敏感文件到外部接口、脱离竞赛项目上下文进行通用聊天、让模型自由执行 shell 命令、复制 Claude 的品牌、图标、配色、文案或专有视觉资产。

## 7. 项目价值

本项目的核心价值在于：将竞赛项目从自然语言包装转化为可检查、可解释、可复核的项目证据对象。它不是生成漂亮材料，而是发现材料缺口、证据缺口、逻辑矛盾和答辩风险，并生成明确补证任务。
