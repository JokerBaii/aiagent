# Workbench-only 实现约束

本文件记录当前版本的工程边界：用户入口只保留 Qt/QML Workbench，所有历史文本入口要求已移除。

## 1. 禁止事项

- 不再新增文本入口层。
- 不在 QML 中实现扫描、规则、评分、证据匹配或报告拼接。
- 不把 LLM 输出写入最终评分。
- 不覆盖原始项目文件。
- 不生成虚假用户、营收、合作、专利、实验结果或市场数据。

## 2. 入口边界

用户流程从 `apps/contest-workbench/main.cpp` 启动。该入口只负责注册 `CompileController` 并加载 QML。

`CompileController` 可以调用：

- `StagedAuditPipeline`
- `DiffVerifier`
- `MarkdownReporter`
- `JsonReporter`
- `LlmBrain`

`CompileController` 不实现业务规则，只做参数收集、状态转换和 QML 模型桥接。

## 3. 模块边界

```text
contest_core
  数据模型、loader、inventory、text、cpir、claim、evidence、consistency、rules、audit、repair、report

contest_agent
  ToolRegistry、PermissionGate、LifecycleHookManager、ProjectMemory、AuditSessionStore、AgentRuntime、StagedAuditPipeline

contest_llm
  EndpointParser、HttpResponseParser、HttpsJsonClient、LlmBrain、BrainAgentLoop

contest-workbench
  Qt/QML UI、CompileController、结果模型、会话模型
```

## 4. 目录归属和代码清洁硬约束

任何新增代码都必须先判断归属，再创建文件。禁止为了图快把逻辑塞进最近打开的文件。

- 审计、评分、规则、证据匹配、文本抽取、报告生成只能进入 `contest_core` 对应子模块。
- 工具协议、权限门控、hooks、智能体运行时、会话轨迹、项目记忆只能进入 `contest_agent`。
- 大模型请求、OpenAI-compatible endpoint、Brain 单步决策、迭代工具循环和工具 decision 解析只能进入 `contest_llm`。
- QML 只能写界面和状态绑定；`CompileController` 只能做 UI 到 C++ 服务的桥接，并展示 AgentRuntime 输出的结构化事件。
- 测试夹具、临时产物、工作区修订稿不得写入源码目录，必须进入临时目录或 `.workspaces` / `.project-trust` 约定位置。
- 不允许新增万能 `Manager`、`Controller`、`Helper` 大杂烩文件；一个文件只承担一个明确模块职责。
- 不允许因为一个功能很小就跨层调用私有细节；需要跨模块时先设计清晰的数据结构或服务接口。
- 不允许在已有大文件里继续堆功能；当文件职责开始混杂时，优先拆到同模块下的小类或小函数。
- 不允许在 `CompileController` 中手写工具轨迹拼装逻辑；工具决策、观察结果、回合事件和 trace 必须来自 `contest_agent` / `contest_llm` 的明确协议。
- 不允许复制粘贴相近逻辑制造多套规则；应复用模块内已有 helper，或补一个边界清楚的局部抽象。
- 每次新增文件都必须能用一句话说明：属于哪个 target、解决哪个业务点、为什么不能放在已有文件。

如果不确定放在哪里，默认先停下阅读 `03_architecture_and_directory_tree.md` 和本文件，不得凭感觉落文件。

## 5. 验收重点

- fresh configure/build 不生成旧文本入口产物；
- Workbench 构建成功；
- 单元测试通过；
- acceptance 和 quality 通过；
- package 中只包含 Workbench、规则包、示例、docs、tools 和 README；
- 全项目文档不得重新要求旧入口层。
